# 工单体验改进计划

本文档记录 Haruki Toolbox 工单系统的轻量改进计划。

项目定位是民间非盈利工具站，因此目标不是建设完整企业级 ITSM / CRM 系统，而是在维护成本可控的前提下，让用户前台和管理员后台都足够顺手、清晰、可靠。

---

## 1. 当前判断

现有实现已经具备一个最小工单闭环：

- 用户可创建工单、查看列表、查看详情、追加消息、关闭工单
- 管理员可查看所有工单、查看详情、回复用户、写内部备注、变更状态、分配处理人
- 后端已有 `tickets` 和 `ticket_messages` 两类核心数据

但从体验上看，当前更像“带标题和状态的客服聊天串”，而不是一个能帮助管理员快速判断下一步动作的工单界面。

另外，前后端目前存在明显契约不一致：

- 后端状态：`open`、`pending_user`、`pending_admin`、`resolved`、`closed`
- 前端状态：`open`、`in_progress`、`resolved`、`closed`
- 后端优先级：`low`、`normal`、`high`、`urgent`
- 前端优先级：`low`、`normal`、`medium`、`high`、`urgent`

这会导致后端返回的 `pending_user` / `pending_admin` 在前端被归一化成默认状态，也会导致前端提交 `medium` 或 `in_progress` 时后端拒绝。

---

## 2. 改进原则

- 保持轻量：不引入复杂 SLA、自动升级、多团队队列、满意度等企业客服系统能力。
- 优先修契约：先保证前后端状态、优先级、响应字段一致。
- 前台清楚：用户能看懂工单是否还在等管理员、是否需要自己补充信息、是否已解决。
- 后台高效：管理员列表能快速找到“待处理”“未分配”“紧急/高优先级”“我的工单”。
- 不把聊天体验完全抹掉：对小项目来说，沟通记录用聊天式展示是合理的，但需要补足工单属性和操作入口。
- 文档和测试同步：公开接口、状态语义、前端字段变更需要同步文档与最小测试。

---

## 3. 分阶段计划

### 阶段 0：修正前后端契约

这是最高优先级，属于 bug 修复。

后端保持当前状态语义：

| 状态 | 含义 |
|------|------|
| `open` | 新建或刚打开 |
| `pending_admin` | 等管理员处理 |
| `pending_user` | 等用户回复 |
| `resolved` | 已解决，可由用户确认关闭或管理员关闭 |
| `closed` | 已关闭，不再接受普通回复 |

需要调整：

- 前端 `TicketStatus` 增加 `pending_user`、`pending_admin`，移除或兼容 `in_progress`
- 前端状态文案改成“待管理员处理”“待用户回复”等更明确的描述
- 前端 `TicketPriority` 移除 `medium`，或后端明确支持 `medium`；建议先移除 `medium`，减少迁移成本
- 前端创建、筛选、状态更新选项必须只提交后端接受的值
- API normalize 层不要把合法后端状态降级成 `open`

验收标准：

- 用户和管理员列表能正确显示 `pending_admin` / `pending_user`
- 管理员状态选择不会提交后端不支持的值
- 用户创建工单不会提交 `medium`
- 前端类型、i18n、显示 helper、筛选选项保持一致

### 阶段 1：前台用户体验整理

目标是让用户明确知道“现在该谁行动”。

建议调整：

- 工单列表卡片突出状态：
  - “待管理员处理”
  - “管理员已回复，等待你补充”
  - “已解决”
  - “已关闭”
- 详情页顶部增加一块简洁摘要：
  - 工单编号
  - 分类
  - 优先级
  - 当前状态
  - 最近更新
- 保留消息气泡，但弱化“即时聊天”的暗示：
  - 输入框文案从“输入消息”调整为“补充说明或回复管理员”
  - 已关闭时明确说明不能继续回复
  - `resolved` 时允许用户关闭，或后续再决定是否允许继续补充
- 创建页增加轻量提示：
  - 分类和优先级不需要过度复杂
  - 引导用户提供错误截图、账号 ID、服务器、相关链接等信息，但先不实现附件也可以

验收标准：

- 用户进入详情页后能在 3 秒内看懂工单状态
- 用户能区分“等管理员”和“等我回复”
- 移动端卡片和详情页不拥挤、不遮挡操作按钮

### 阶段 2：后台管理员工作台整理

目标是让管理员少点来点去，列表就能判断优先级，详情页能集中处理。

列表建议：

- 增加快速筛选：
  - 全部
  - 待管理员处理
  - 待用户回复
  - 未分配
  - 我的工单
  - 高/紧急
- 列表字段保留轻量：
  - 标题
  - 状态
  - 优先级
  - 创建者
  - 处理人
  - 最近更新
- 可以考虑增加 `lastMessagePreview` / `lastMessageAt`，如果后端查询成本可接受，也可以先由详情页承担，不急着做冗余字段。

详情页建议：

- 顶部保留标题、状态、优先级、创建者。
- 操作区固定展示：
  - 修改状态
  - 分配处理人
  - 回复用户
  - 内部备注
- 侧边或顶部展示用户上下文入口：
  - 用户 ID / 用户名
  - 跳转用户详情
  - 相关游戏账号或上传记录可以后续再接，不作为第一轮目标
- 内部备注继续只对管理员可见。

验收标准：

- 管理员能一键筛出“待管理员处理”的工单
- 管理员能快速分辨未分配工单
- 管理员回复后状态自动进入 `pending_user`
- 用户回复后状态自动进入 `pending_admin`
- 内部备注不会暴露给用户接口

### 阶段 3：轻量邮件通知

这一阶段补足最基本的异步提醒能力，仍然保持轻量。

目标：

- `role = admin` / `role = super_admin` 的用户可以自行开关是否接收工单邮件通知。
- 用户在自己的工单收到管理员公开回复时，应该收到邮件提醒。
- 邮件通知是辅助能力，发送失败不能影响工单创建、回复、关闭等主流程。

管理员通知偏好：

- 建议在 `users` 表增加一个布尔字段，例如 `ticket_email_notifications_enabled`。
- 该字段只对 `admin` / `super_admin` 有实际意义，普通用户即使字段存在也不参与管理员通知。
- 默认值建议为 `false`，避免升级后突然给所有管理员发邮件。
- 后台“个人设置”或“管理员用户详情”里提供开关。
- 如果管理员被封禁或角色被降级为普通用户，不再接收管理员工单通知。

邮件触发规则：

| 事件 | 收件人 | 说明 |
|------|--------|------|
| 用户创建新工单 | 已开启工单通知的 `admin` / `super_admin` | 可作为后台待处理提醒 |
| 用户追加公开回复 | 已开启工单通知的 `admin` / `super_admin` | 已分配工单可以优先通知处理人，未分配则通知所有已开启管理员 |
| 管理员追加公开回复 | 工单创建者 | 提醒用户查看管理员回复 |
| 管理员追加内部备注 | 不发用户邮件 | 内部备注只对管理员可见 |
| 状态变更 / 分配变更 | 暂不发邮件 | 先避免通知过多，后续按实际反馈再补 |

邮件内容建议：

- 标题包含工单编号和简短主题，例如：`[Haruki Toolbox] 工单有新回复：TK-...`
- 正文包含：
  - 工单编号
  - 工单主题
  - 当前状态
  - 回复摘要，限制长度并去除多余空白
  - 前端详情页链接
- 不在邮件里放完整敏感内容；长文本只截断展示。

发送策略：

- 数据库事务先提交，提交成功后再发送邮件。
- 邮件发送失败只记录日志，不回滚工单操作。
- 第一版可以同步发送；如果后续发现请求变慢，再改成后台任务或队列。
- 必须避免给回复发起者自己发邮件。
- 管理员内部备注、关闭操作、纯分配操作默认不触发用户邮件。

验收标准：

- 管理员可以打开/关闭自己的工单邮件通知。
- 普通用户无法通过该开关让自己进入管理员通知列表。
- 新工单或用户回复会通知已开启通知的管理员。
- 管理员公开回复会通知工单创建者。
- 管理员内部备注不会通知用户。
- 邮件发送失败时，API 仍然返回工单操作成功，并在日志中记录失败原因。

### 阶段 4：轻量证据与上下文能力

这一阶段按实际需求决定是否做，不作为第一轮必须项。

可选能力：

- 附件：
  - 先支持图片或文本日志
  - 限制大小、数量和类型
  - 前端展示为消息的一部分
- 关联对象：
  - 上传问题可关联上传记录
  - 账号问题可关联用户详情
  - Webhook / OAuth2 问题可粘贴相关 ID
- 系统事件消息：
  - 分配处理人
  - 状态变更
  - 关闭工单
  - 这些可以作为 `sender_role = system` 的消息展示给管理员，是否展示给用户可以单独控制

建议先不要上完整事件表，除非后续需要精确审计或复杂统计。

### 阶段 5：前端轻量提醒

如果后台处理频率变高，再补前端内的轻量提醒。

优先级从低成本开始：

- 前端后台导航显示待处理数量
- 管理员列表定时刷新或手动刷新
- 用户进入前台时提示有管理员回复

暂不建议第一轮引入 WebSocket / SSE，除非站点已经有统一实时通道。

---

## 4. 暂不做的内容

以下能力对当前项目可能维护成本大于收益，暂不作为目标：

- 多客服团队与复杂队列
- SLA 计时、逾期升级、排班
- 客服绩效、满意度、服务报表
- 宏回复 / 知识库 / 自动分流
- 多渠道接入，例如邮件转工单、IM 转工单
- 复杂权限矩阵

---

## 5. 推荐实施顺序

1. 修前后端状态和优先级契约。
2. 更新前端状态文案、筛选项、badge、normalize 逻辑。
3. 整理用户详情页，让“当前等待谁”更明显。
4. 整理管理员列表，增加轻量快速筛选。
5. 整理管理员详情页，把状态、处理人、回复、内部备注放成稳定操作区。
6. 增加管理员工单邮件通知开关。
7. 增加工单回复邮件通知，管理员公开回复通知用户，用户创建/回复通知已开启通知的管理员。
8. 视真实反馈决定是否加入附件、系统事件消息、前端通知计数。
