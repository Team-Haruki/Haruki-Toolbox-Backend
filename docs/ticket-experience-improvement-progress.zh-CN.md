# 工单体验改进进度

本文档记录 `docs/ticket-experience-improvement-plan.zh-CN.md` 的当前落地进度，按已经完成、正在推进、后续待做三个层次整理，方便继续迭代。

---

1. **管理员工单列表快速筛选接口**
2. **管理员工单列表轻量摘要字段**
3. **管理员工单详情消息发送者名称**
4. **管理员工单邮件通知开关**
5. **工单邮件通知投递**
6. **管理员内部系统事件消息**
7. **前端管理员工单列表接入和通知开关入口**
8. **前端用户工单列表和详情状态展示优化**
9. **前端管理员工单详情页操作区整理**
10. **前端用户创建工单页轻量提示**
11. **前端管理员工单列表手动刷新入口**
12. **前端后台工单待处理计数**
13. **前端用户待回复计数**

涉及接口：

- `GET /api/admin/tickets`

新增查询参数：

| 参数 | 含义 | 后端行为 |
|------|------|----------|
| `quick_filter=pending_admin` | 待管理员处理 | 等价于默认补上 `status=pending_admin` |
| `quick_filter=pending_user` | 待用户回复 | 等价于默认补上 `status=pending_user` |
| `quick_filter=unassigned` | 未分配 | 默认筛选 `assignee_admin_id IS NULL` |
| `quick_filter=mine` | 我的工单 | 默认筛选 `assignee_admin_id = 当前管理员 ID` |
| `quick_filter=high_or_urgent` | 高 / 紧急 | 默认筛选 `priority IN (high, urgent)` |
| `quick_filter=all` 或留空 | 全部 | 不追加快捷筛选条件 |

补充规则：

- `q`、`page`、`page_size` 仍可与 `quick_filter` 一起使用。
- 如果显式传了 `status`、`priority`、`assignee_admin_id`，则显式参数优先，`quick_filter` 只作为快捷默认值。
- 非法 `quick_filter` 会返回 `400 Bad Request`。

本轮同时补了对应单元测试，覆盖：

- `quick_filter` 合法值和非法值解析
- “我的工单 / 未分配 / 高或紧急”快捷筛选展开逻辑
- 显式筛选参数覆盖快捷筛选默认值的行为

列表摘要字段新增内容：

| 字段 | 含义 |
|------|------|
| `assigneeAdminName` | 处理人名称 |
| `lastMessageAt` | 最后一条消息时间 |
| `lastMessageSenderRole` | 最后一条消息发送方角色 |
| `lastMessagePreview` | 最后一条消息的轻量预览 |
| `lastMessageInternal` | 最后一条消息是否为内部备注 |

摘要字段规则：

- 管理员列表查询会预加载每张工单的最后一条消息，用于构造列表摘要。
- 消息预览会压缩多余空白，并在超长时截断为固定长度，避免列表被长文本撑开。
- 处理人名称会和创建人名称一起返回，减少前端额外查表拼接成本。
- 详情接口和状态 / 分配更新后的返回体也会带上同样的摘要字段，保持结构一致。

详情消息字段新增内容：

| 字段 | 含义 |
|------|------|
| `senderUserName` | 当前消息发送者名称 |

补充规则：

- 管理员工单详情接口会为消息发送者补齐 `senderUserName`。
- 管理员追加消息接口返回的单条消息对象也会带 `senderUserName`，便于前端直接插入消息列表。
- `senderUserID` 为空的系统消息仍保持无名称返回。

补充规则：

- 管理员可以通过 `admin/me` 接口读取和更新自己的工单通知开关。

- `external/oathkeeper/access-rules.yml` 已有 `/api/admin/<.*>` 通配规则，因此这次新增 `admin/me` 子路由不需要额外改规则。

工单邮件通知投递新增内容：

涉及触发点：

| 触发点 | 收件人 | 说明 |
|--------|--------|------|
| 用户创建新工单 | 已开启通知的 `admin` / `super_admin` | 排除被封禁管理员和普通用户 |
| 用户追加公开回复 | 优先通知已开启通知的当前处理人；无可用处理人时通知所有已开启通知的管理员 | 排除回复发起者 |
| 管理员追加公开回复 | 工单创建者 | 内部备注不会发给用户 |

补充规则：

- 邮件发送在数据库事务提交成功后执行。
- 邮件发送失败只写日志，不回滚创建工单或追加回复的主流程。
- 邮件内容包含工单编号、主题、当前状态、回复摘要和前台工单详情链接。
- 回复摘要会压缩空白、截断长文本，并进行 HTML escape，避免把完整长文本直接塞进邮件。
- 用户回复后的邮件状态按提交后的 `pending_admin` 展示；管理员公开回复后的邮件状态按提交后的 `pending_user` 展示。
- 邮件详情链接使用 `user_system.frontend_url` 拼出 `/tickets/{ticketId}`，面向普通用户和管理员都可直接打开前台详情。
- 当前第一版仍为同步发送；如果线上感觉接口响应变慢，再改成后台任务或队列。

管理员内部系统事件消息新增内容：

涉及触发点：

| 触发点 | 系统消息 | 展示范围 |
|--------|----------|----------|
| 管理员变更状态 | `Status changed: Open -> Resolved` | 管理员详情和管理员列表最近动态 |
| 管理员变更处理人 | `Assignee changed: Unassigned -> name (id)` | 管理员详情和管理员列表最近动态 |

补充规则：

- 系统事件复用现有 `ticket_messages` 表，不新增事件表。
- 系统事件写入时使用 `sender_role = system`，并固定 `internal = true`。
- 用户详情接口继续只返回 `internal = false` 的消息，因此这些内部操作记录不会展示给普通用户。
- 无实际变化的重复状态更新或重复指派不会追加系统消息，避免消息流被空操作刷屏。
- 状态事件使用可读状态标签，处理人事件会优先记录用户名称并保留用户 ID。
- 管理员列表“最近动态”会把系统事件纳入最后一条动态摘要，方便列表上直接看到刚发生的状态或指派变化。

## 2. 当前效果

当前已完成的后端改动，主要解决了三类基础问题：

1. 管理员列表和详情页如何更快判断下一步动作
2. 管理员工单邮件通知偏好如何先落库并自助开关
3. 工单有新建或公开回复时如何做最小邮件提醒

落地后，前端在不拼装多组状态 / 优先级 / assignee 条件的前提下，就可以直接通过一个稳定的 `quick_filter` 参数拿到以下常见视图：

- 待管理员处理
- 待用户回复
- 未分配
- 我的工单
- 高 / 紧急

同时，管理员列表项本身已经具备：

- 创建人名称
- 处理人名称
- 最后一条消息时间
- 最后一条消息角色
- 最后一条消息预览
- 最后一条消息是否为内部备注

管理员详情消息也已经具备：

- 发送者 ID
- 发送者名称
- 发送者角色
- 是否为内部备注

管理员通知偏好也已经具备：

- 数据库字段持久化
- `admin/me` 读取接口
- `admin/me` 更新接口

邮件通知现在已经具备：

- 新工单通知已开启通知的管理员
- 用户公开回复通知处理人或管理员组
- 管理员公开回复通知工单创建者
- 内部备注不通知用户
- 邮件发送失败不影响接口成功返回

管理员内部系统事件现在已经具备：

- 状态变更会写入管理员可见的系统消息，并使用可读状态标签
- 处理人变更会写入管理员可见的系统消息，并优先显示处理人名称
- 系统事件消息不会出现在普通用户详情接口
- 管理员列表最近动态会显示系统事件摘要

前端管理员列表现在已经具备：

- 正确显示 `pending_admin` / `pending_user`
- 创建和筛选不再使用后端不支持的 `medium`
- 一键筛选待处理、未分配、我的工单和高优先级工单
- 列表直接展示最近动态摘要和内部备注标记
- 管理员可在工单管理页打开或关闭自己的工单邮件通知

用户前台现在已经具备：

- 用户能在列表直接看到工单编号、状态和当前行动方提示
- 用户能在详情页顶部看到工单摘要，而不需要从消息串里推断上下文
- 用户能区分“等管理员处理”“等我补充”“已解决可关闭”“已关闭”
- 用户可以在管理员标记 `resolved` 后自行关闭工单

管理员详情页现在已经具备：

- 管理员能先看到工单摘要，再进行处理操作
- 状态变更和处理人分配被集中在同一个操作区
- 公开回复和内部备注在输入区有更明确的模式提示
- 沟通记录区明确区分用户、管理员、系统和内部备注

用户创建页现在已经具备：

- 用户能在提交前看到每个字段该如何填写
- 用户更容易把账号、服务器、复现步骤和错误提示一次写清楚
- 紧急优先级的使用语义更明确

管理员列表现在还具备：

- 管理员可以在保留当前筛选条件的情况下手动刷新列表
- 处理新邮件通知或线下提醒后，不需要刷新整个页面

后台导航现在还具备：

- 工单管理入口能显示等待管理员处理的工单数量
- 管理员不进入列表也能感知是否有待处理工单
- 计数会轻量自动刷新，不需要管理员反复切页

前台导航现在还具备：

- “我的工单”入口能显示等待用户回复的工单数量
- 用户不进入列表也能感知是否有管理员回复需要处理
- 计数会轻量自动刷新，不需要引入实时通道

这部分对应原计划中的：

- “管理员能一键筛出待管理员处理的工单”

- “列表就能判断优先级，详情页能集中处理”中的列表判断基础
- “详情页能集中处理”中的消息展示基础
- “管理员可以打开/关闭自己的工单邮件通知”
- “新工单或用户回复会通知已开启通知的管理员”
- “管理员公开回复会通知工单创建者”
- “管理员内部备注不会通知用户”
- “邮件发送失败时，API 仍然返回工单操作成功”
- “系统事件消息”中的状态变更和处理人变更
- “前端类型、i18n、显示 helper、筛选选项保持一致”
- “前端管理员工单列表接入 quick_filter 和新增摘要字段”
- “前端增加管理员自己的工单邮件通知开关入口”
- “用户进入详情页后能在 3 秒内看懂工单状态”

- “管理员详情页固定操作区整理”
- “详情页能集中处理”中的状态、处理人、回复、内部备注操作区
- “创建页增加轻量提示”

## 3. 还没开始或尚未完成的部分

本轮**还没有**处理以下内容：

- 附件
- 更实时的统一通知通道，例如通知中心、WebSocket / SSE

也就是说，当前完成的是**管理员工作台基础接口能力**、**后端邮件通知闭环**、**管理员内部系统事件消息**、**管理员列表前端接入**、**用户前台状态展示整理**、**管理员详情页整理**和**轻量前端计数提醒**，不是整套工单体验改造。附件和更实时的统一通知通道仍属于后续可选增强。

---

## 4. 建议下一步

下一步最顺手的衔接项有三个方向：

1. 视真实需求决定是否补附件或统一通知中心。
2. 如果用户确实经常需要提交图片或日志，再单独设计附件能力。
3. 如果后续需要秒级实时体验，再考虑 WebSocket / SSE；当前 60 秒轻量刷新已经足够覆盖大多数小项目工单场景。

如果继续做后端，可以考虑把附件作为下一阶段；但以当前项目定位看，它可以等真实需求更明确后再做。
